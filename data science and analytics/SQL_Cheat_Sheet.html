2SORTING & LIMITING
-- ORDER BY (default ASC) SELECT * FROM products ORDER BY unit_price DESC; -- Pagination SELECT * FROM products ORDER BY unit_price DESC LIMIT 10 OFFSET 10;
Multi-column sort: ORDER BY category ASC, price DESC — first column is the primary sort, later columns break ties.
⚠️ Always pair LIMIT with ORDER BY — otherwise "top N" is not deterministic between runs.
3AGGREGATIONS Σ
SELECT region, COUNT(*) AS cnt, SUM(revenue) AS total_sales, AVG(revenue) AS avg_sales FROM orders GROUP BY region HAVING SUM(revenue) > 100000;
FunctionNULL behavior
COUNT(*)Counts ALL rows (incl. NULLs)
COUNT(col)Ignores NULLs in that column
SUM / AVG / MIN / MAXIgnore NULLs
WHERE filters rows BEFORE grouping. HAVING filters GROUPS afterward — and can use aggregate functions.
4JOINS
SELECT o.order_id, c.customer_name FROM orders o INNER JOIN customers c ON o.customer_id = c.customer_id; -- LEFT JOIN keeps all rows from orders ... LEFT JOIN customers c ON ...
TypeKeeps
INNER JOINOnly rows matched in both tables
LEFT JOINAll left rows + matches (NULL if none)
RIGHT JOINAll right rows + matches
FULL OUTER JOINAll rows, both sides (no MySQL support)
Anti-join: LEFT JOIN ... WHERE right.id IS NULL — finds "no match" rows (e.g. customers with no orders).

5SUBQUERIES & CTEs
-- Subquery in WHERE (scalar) SELECT * FROM employees WHERE salary > (SELECT AVG(salary) FROM employees);
-- CTE (WITH clause) WITH region_sales AS ( SELECT region, SUM(revenue) AS rev FROM orders GROUP BY region ) SELECT * FROM region_sales WHERE rev > 100000;
A CTE names a subquery so you can reference it like a table — it only exists for that one query. CTEs can chain (reference earlier CTEs in the same WITH clause).
6WINDOW FUNCTIONS 📊
SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) AS rn, RANK() OVER (ORDER BY salary DESC) AS rnk, DENSE_RANK() OVER (ORDER BY salary DESC) AS drnk, SUM(salary) OVER (PARTITION BY dept_id ORDER BY salary) AS running_total FROM employees;
RANK leaves gaps after ties (1,2,2,4); DENSE_RANK doesn't (1,2,2,3). PARTITION BY restarts the calc per group WITHOUT collapsing rows. Also useful: LAG, LEAD, NTILE.
7CONDITIONAL LOGIC
SELECT name, salary, CASE WHEN salary >= 100000 THEN 'High' WHEN salary >= 50000 THEN 'Medium' ELSE 'Low' END AS salary_band FROM employees;
-- CASE inside an aggregate SELECT region, COUNT(CASE WHEN status='Cancelled' THEN 1 END) AS cancelled FROM orders GROUP BY region;
Checked top-to-bottom — first match wins. ELSE is optional (omitted = NULL for non-matches).
8DATA MANIPULATION (DML)
INSERT INTO table_name (col1, col2) VALUES (val1, val2); UPDATE table_name SET col1 = val1 WHERE condition; DELETE FROM table_name WHERE condition;
⚠️ ALWAYS test your WHERE clause as a SELECT first! No WHERE on UPDATE/DELETE = entire table affected.

12SET OPERATIONS ⇄
SELECT col FROM a UNION SELECT col FROM b; -- dedups rows
VariantBehavior
UNION ALLKeep duplicates (faster)
INTERSECTRows in both queries
EXCEPTRows in 1st, not 2nd (A minus B)
⚠️ Same column count & compatible types required. MySQL lacks INTERSECT/EXCEPT before 8.0.31.
13KEYS & INDEXES 🔑
ConceptWhat it does
PRIMARY KEYUnique row ID, not NULL
FOREIGN KEYReferences PK in another table
INDEXSpeeds up lookups on a column
CREATE INDEX idx_orders_date ON orders(order_date);
Index = faster reads, slower writes, extra storage. Index columns used in WHERE / JOIN / ORDER BY.
14COMMON TIPS 💡
NULL Handling
COALESCE(phone,'N/A') AS phone
NULL = NULL is NULL, not TRUE. Use IS NULL / IS NOT NULL.
